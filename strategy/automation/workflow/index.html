    評測流程自動化

    將 LLM 評測整合到 GitLab CI/CD 流程中，確保每次模型更新都經過嚴格驗證，實現品質的持續保障。

    自動化評測解決的核心痛點

    在 LLM 開發迭代中，團隊常面臨以下挑戰：

    痛點一：評測與開發脫節

    graph LR
    subgraph "傳統模式 ❌"
        D1[開發迭代] --> D2[手動評測]
        D2 --> D3[發現問題]
        D3 --> D4[已上線...]
    end

    subgraph "自動化模式 ✅"
        A1[開發迭代] --> A2[自動評測]
        A2 --> A3{問題?}
        A3 -->|有| A4[阻止合併]
        A3 -->|無| A5[允許上線]
    end

    真實案例

    團隊更新了 Prompt 以改善回答風格，但無意間導致 Faithfulness 下降 15%。 由於缺乏自動化評測，問題在上線一週後才被用戶投訴發現。

    痛點二：指標退化難以察覺

    當模型經過多次迭代，缺乏系統性追蹤時：

    迭代 變更內容 Faithfulness Relevancy 累積效應
    v1.0 基線版本 0.92 0.88 -
    v1.1 優化回答格式 0.90 (-2%) 0.89 開始退化
    v1.2 增加安全過濾 0.88 (-2%) 0.87 持續惡化
    v1.3 調整 Temperature 0.85 (-3%) 0.85 嚴重退化

    自動化評測通過 Baseline 比較機制，在每次變更時檢測退化。

    痛點三：安全問題反應遲緩

    graph TB
    subgraph "風險場景"
        R1[Prompt 變更]
        R2[繞過安全過濾]
        R3[毒性回應上線]
        R4[用戶投訴/PR 危機]
    end

    subgraph "自動化防護"
        A1[Prompt 變更]
        A2[Safety Gate 攔截]
        A3[阻止部署]
        A4[開發者修復]
    end

    R1 --> R2 --> R3 --> R4
    A1 --> A2 --> A3 --> A4

    自動化的 Safety Gate 確保每次部署前都通過 偏見檢測 與毒性測試。

    痛點四：評測成本與效率矛盾

    評測類型 樣本數 耗時 成本
    完整評測 10,000 2 小時 $50
    冒煙測試 100 3 分鐘 $0.5

    分層評測策略 解決此矛盾——日常開發用冒煙測試快速回饋，關鍵節點執行完整評測。

    核心技術與概念

    1. 持續整合/持續交付 (CI/CD)

    flowchart LR
    subgraph CI["持續整合 CI"]
        A[代碼提交] --> B[自動建置]
        B --> C[自動測試]
    end

    subgraph CD["持續交付 CD"]
        C --> D[部署 Staging]
        D --> E[驗收測試]
        E --> F[部署 Production]
    end
    階段 在 LLM 評測中的應用
    CI - 建置 驗證 Prompt 模板格式、配置語法
    CI - 測試 執行評測套件，計算 評估指標
    CD - Staging 在隔離環境驗證端到端行為
    CD - Production 漸進式發布 + 線上監控

    2. 閘控機制 (Gating)

    閘控是自動化評測的核心，決定變更是否可以推進：

    flowchart TD
    START[評測結果] --> G1{Safety Gate}
    G1 -->|失敗| BLOCK1[❌ 阻止]
    G1 -->|通過| G2{Quality Gate}
    G2 -->|失敗| REVIEW[⚠️ 人工審核]
    G2 -->|通過| G3{Baseline Gate}
    G3 -->|退化| REVIEW
    G3 -->|進步| PASS[✅ 允許部署]

    REVIEW -->|批准| PASS
    REVIEW -->|拒絕| BLOCK2[❌ 阻止]

    三層閘控設計：

    閘控層 檢查內容 失敗處理
    Safety Gate 毒性、偏見、PII 洩露 必須阻止，不可覆蓋
    Quality Gate Faithfulness、Relevancy 等 低於閾值需審核
    Baseline Gate 與 Production 版本比較 退化需審核或阻止

    3. Baseline 比較機制

    確保模型品質只進不退的關鍵技術：

    flowchart TB
    subgraph "Baseline 管理"
        BL[Production Baseline]
        STORE[(版本化儲存)]
    end

    subgraph "評測流程"
        NEW[候選版本]
        EVAL[執行評測]
        CMP[與 Baseline 比較]
    end

    BL --> STORE
    STORE --> CMP
    NEW --> EVAL --> CMP

    CMP --> DECISION{勝過 Baseline?}
    DECISION -->|是| PROMOTE[✅ 部署 & 更新 Baseline]
    DECISION -->|否| REJECT[❌ 拒絕部署]

    PROMOTE --> STORE

    比較維度：

    • 整體分數：加權後的綜合指標，參考 指標組合與權重
    • 關鍵指標：Faithfulness、Safety 等不允許任何退化
    • 次要指標：可接受小幅退化（<2%），但整體需正向

    4. 漸進式發布 (Progressive Rollout)

    離線評測無法完全模擬真實用戶行為，因此需要漸進式驗證：

    flowchart LR
    STG[Staging<br>100% 測試流量] -->|手動批准| CAN[Canary<br>5% 真實流量]
    CAN -->|監控 1hr| P25[25%]
    P25 -->|監控 2hr| P50[50%]
    P50 -->|監控 4hr| FULL[100%]

    CAN -.->|異常| ROLL[自動回滾]
    P25 -.->|異常| ROLL
    P50 -.->|異常| ROLL

    監控指標：

    • 效能：Latency P95、錯誤率
    • 品質：用戶評分、重試率
    • 安全：毒性回報、異常偵測

    評測指標的自動化應用

    將 評估指標體系 整合到 CI/CD：

    指標分層與閾值

    指標類別 指標 CI 閾值 說明
    安全性 Toxicity < 0.01 必須通過，參考 Safety Gate
    安全性 Bias Score < 0.05 必須通過
    一致性 Faithfulness ≥ 0.90 RAG 系統核心指標
    相關性 Answer Relevancy ≥ 0.85 回答切題程度
    效能 Latency P95 < 2000ms 用戶體驗保障

    動態閾值調整

    隨著模型成熟，可逐步提高閾值：

    graph LR
    V1[v1.0<br>Faithfulness ≥ 0.85] --> V2[v2.0<br>Faithfulness ≥ 0.88]
    V2 --> V3[v3.0<br>Faithfulness ≥ 0.90]
    V3 --> V4[v4.0<br>Faithfulness ≥ 0.92]

    關鍵 Job 設計

    # .gitlab-ci.yml 核心配置

stages:
  - preflight    # 快速檢查
  - evaluate     # 深度評測
  - safety       # 安全閘控
  - deploy       # 部署階段

# 冒煙測試：快速回饋
smoke_test:
  stage: preflight
  script:
    - python -m eval.smoke --samples 100
  timeout: 3 minutes

# 完整評測：計算所有指標
full_evaluation:
  stage: evaluate
  script:
    - python -m eval.run --config eval_config.yaml
  artifacts:
    paths: [results/]
  timeout: 30 minutes

# Baseline 比較：確保進步
baseline_compare:
  stage: evaluate
  needs: [full_evaluation]
  script:
    - python -m eval.compare --baseline production
  allow_failure: false

# 安全閘控：不可妥協
safety_gate:
  stage: safety
  script:
    - python -m eval.safety --strict
  allow_failure: false  # 必須通過

# 漸進部署
deploy_canary:
  stage: deploy
  script:
    - ./deploy.sh --canary 5%
  when: manual
  environment: production-canary

    MR 驅動的評測流程

    sequenceDiagram
    participant Dev as 開發者
    participant MR as Merge Request
    participant CI as GitLab CI
    participant Rev as 審核者

    Dev->>MR: 創建/更新 MR
    MR->>CI: 觸發 Pipeline

    CI->>CI: 冒煙測試 (3 min)
    CI->>CI: 完整評測 (30 min)
    CI->>CI: Baseline 比較
    CI->>CI: Safety Gate

    alt 全部通過 & 優於 Baseline
        CI-->>MR: ✅ 通過<br>📈 指標改善報告
        MR->>Rev: 請求審核
        Rev->>MR: Approve
    else Safety 失敗
        CI-->>MR: ❌ 阻止合併<br>🚨 安全問題詳情
    else 劣於 Baseline
        CI-->>MR: ⚠️ 需人工審核<br>📉 退化分析
        MR->>Rev: 需額外審核
    end

    與 Responsible AI 的整合

    自動化評測是實踐 Responsible AI 的關鍵機制：

    Responsible AI 原則 自動化實踐
    公平性 每次部署前執行 公平性測試
    偏見緩解 Safety Gate 包含 偏見檢測
    可解釋性 評測報告包含 歸因分析
    持續監控 線上指標持續追蹤 Responsible AI 合規

    最佳實踐總結

    關鍵原則

    原則 實踐方式
    只進不退 Baseline 比較機制，退化必須審核
    安全優先 Safety Gate 不可覆蓋
    快速回饋 分層測試：冒煙 3min + 完整 30min
    漸進驗證 Canary 5% → 25% → 50% → 100%
    全程追蹤 指標版本化，可追溯品質演進

    常見陷阱

    • ❌ 只關注準確率，忽略安全指標
    • ❌ 閾值設太低，失去防護意義
    • ❌ 跳過 Canary，直接全量發布
    • ❌ 只看絕對值，不做 Baseline 比較
